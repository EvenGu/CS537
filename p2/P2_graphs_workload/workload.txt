Description:
// GRAPH 1 //

This is the start up of the xv6 which start with a single process
init, followed by the starting of the first sh. After the OS starts
I run 'ls'. This workload shows simple consecutive execution of
processes in sequential order.

The expected behaviour is quite simple, it should run each process
through the correct number of ticks at each priority.

// GRAPH 2 //

From the shell running 'ls &; mkdir stillTesting &; echo this &;'

This executes several commands in parallel and we expect to see
them being initialized and ran 1 priority at a time, then RR in
the final priority queue.

There appears to be something awkward happening when 'ls' gave up
the CPU and went to 'sleeping' state after 4 ticks, this was 
unusual but the scheduler correctly moved on to another process
in the queue since ls was not 'RUNNABLE' at the time.

// GRAPH 3 //

Here we run three 'tester' processes in parallel each looping in
a loop for 1,000,000 iterations. Roughly each program should run
about the same amount of time.

It is clearly noticeable that each pocess ran for 16 or 17 ticks,
which means they shared the resources equally. Furthermore we see
that each priority trumped as described in the spec. Since all
processes can't be spawned EXACTLY at the same time and set to
'RUNNABLE' we see the mixed execution between p0,p1 and p3 in the
first few ticks.

For the remainder of the execution we see the three processes run
in RR in the order they left the previous priority. Since neither
process every had any I/O or fell asleep, the order was never
switched or interrupted.

Finally the init/shell runs to clean up zombie processes.

// GRAPH 4 //

The final graph is probably most interesting and demonstrates the
functioning MLFQ most accurately.

We run the command 'tester 1000000 fork &; tester 1000000 fork &;'.
Giving the third parameter the tester program forks at 500,000
iterations - or roughly half way. This means during run time we will
introduce two new process in the lowest priority.

As the data shows - and corresponding graph - we start off by creating
the first new processes and gradually move up through the queues of
correct priority.

At a certain point in the third priority one of the programs forks and
we see it trump the processes in the highest priority. As it returns,
the execution of the process that was forked continues where it left
off. This then follows the correct order of the processes as they
entered the queue.

At some point we also get the second fork which follows a similar
pattern which runs through the priorities. We see that the forked
processes roughly execute for half the number of ticks (makes sense)
,and finish before the original processes as they have access to
the lower priorities. Finally the init/sh processes run to clean up
any zombie processes.
